# Codex-Claude AutoFlow v1.0.0

首个公开可用版本，核心目标是让 `Codex -> Claude -> Codex` 在本地工程里形成一条可重复、可恢复、可审查的全自动协作链路。

## 本版亮点

- 支持 `Codex` 先出计划，`Claude` 按计划直接改代码，最后再由 `Codex` 审查和终审。
- 支持多个允许修改的代码目录，适合主代码目录和少量入口文件分散的工程。
- 提供 Windows 图形界面，不需要手动拼命令行参数。
- 支持任务标识、任务历史、历史任务恢复、暂停后继续。
- 支持终审未通过后的“再修再审”循环。
- 支持任务成功后由 `Codex` 自动提交一次 git commit。

## 这次重点改进

### 1. 自动提交不再把“生成了说明文件”误判成“提交成功”

现在提交阶段不会只看有没有 `06_codex_commit.md`，而是会进一步检查仓库状态，确认是否真的创建了新的 git commit。

### 2. commit 阶段不再因为审批弹窗卡住

自动提交阶段已经调整为非交互执行，目标是避免在最终 commit 时再弹出额外审批，影响整条自动化链路。

说明：

- 这并不代表所有 git 问题都会消失。
- 如果本地 `git` 本身有配置问题，例如用户名、邮箱、GPG 签名或仓库策略限制，提交仍然可能失败。
- 失败原因会被写入 `workflow/06_codex_commit.md`。

### 3. 新任务按钮逻辑更清晰

现在选择一个已经有活动 `workflow` 的工程后，界面自动回填上次任务是正常行为，只表示“恢复当前任务状态”，不表示自动开始执行。

按钮逻辑已经明确区分为：

- `开始运行`：在没有活动 `workflow` 时启动一个新任务
- `继续上次任务`：继续当前任务
- `再修再审`：终审被拒绝后继续下一轮
- `新任务草稿`：归档当前 `workflow`，清空任务名称和任务描述，但不会立刻运行

推荐的新任务操作方式：

1. 选择工程目录
2. 点击 `新任务草稿`
3. 重新填写任务标识和任务描述
4. 点击 `开始运行`

## 适用场景

- 本地单片机 / 嵌入式工程
- 需要限制只修改某几个代码目录的项目
- 需要“先计划、再实现、再审查”的自动化改码流程
- 需要保留任务产物和任务历史，便于回溯的项目

## 使用前提

- Windows 系统
- 已安装并登录 `codex` CLI
- 已安装并登录 `claude` CLI

普通使用者只下载 `CodexClaudeWorkflow.exe` 即可，不需要 Python，也不需要完整源码仓库。

## 已知说明

- 选择工程目录后，如果该工程已有活动任务，界面会自动加载当前 `workflow`。
- 自动提交只应对“审批阻塞”问题，不保证绕过本地 git 配置或仓库策略问题。
- `dist/` 更建议作为 GitHub Release 附件发布，而不是直接提交到仓库历史。
